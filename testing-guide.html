Testing interview field guide

Testing, from zero

A plain-English introduction to automated testing — what it is, the kinds of tests, why different parts of the code are tested differently, and exactly how backend and frontend tests usually work in a full-stack application. No prior testing knowledge assumed.

1. Why We Test

A test is just code that checks other code. You write a small program that runs your real function, then asserts — "the answer should be X". If it is, the test passes. If it is not, the test fails loudly and tells you where.

That sounds simple, and it is. The value is not in any single test — it is in having hundreds of them that run in seconds. Concretely, tests buy you:

  • A safety net. You change one function and run the suite. If you accidentally broke something three files away, a test goes red immediately — not next week in production. A bug caught by an old test coming back is called a regression.
  • Confidence to change things. Without tests, every edit is a gamble and you end up afraid of your own code. With tests, you can refactor freely: if the suite is still green, behaviour is unchanged.
  • Executable documentation. A test literally shows how a function is meant to be called and what it returns. It cannot go stale the way a comment can — if it lies, it fails.
  • A design check. Code that is painful to test is usually painful to use. If a test needs ten lines of setup, the function probably does too much.
Mindset You are not testing to prove the code works. You are testing to find out where it doesn't. A test that never fails taught you nothing; a test that catches a real mistake just paid for itself.

2. The Kinds of Tests

"Testing" is not one activity. Tests are categorised mostly by how much of the system they exercise at once. The three you must know:

Unit test

Tests one small piece in isolation — usually a single function — with no database, no network, no web server. You hand it inputs, you check its output. Unit tests are tiny and run in milliseconds.

In a data app, transformation functions are often pure logic: give them a table, they return a new table. Perfect unit-test material — for example tests/test_transformations.py.

Integration test

Tests several pieces working together — for example an API endpoint plus the database plus the service layer, all in one go. Slower than unit tests (there is real wiring involved) but they catch a whole class of bugs unit tests cannot: mismatches between components.

Most of tests/test_auth.py is integration testing: it sends a real HTTP request to the sign-up endpoint and checks that a user row was created and a cookie came back.

End-to-end test (E2E)

Tests the whole system the way a user would — e.g. a browser clicking through the real frontend, which calls the real backend, which hits a real database. The most realistic, but also the slowest and most fragile.

Typical E2E shape A typical full-stack app might keep Playwright E2E tests under e2e/. They drive the real frontend, backend, and a real database. Treat them as high-value journey tests, not as the place to check every tiny rule.

Beyond that "how much" axis, a few named tests describe purpose:

NameWhat it isExample
Regression test A test written specifically to lock in a bug fix so the bug can never silently return. test_save_project.py — proves a save never corrupts the original uploaded file.
Smoke test A quick, shallow "does it even start / respond at all" check. e.g. asserting GET /projects/recent returns 200 for a logged-in user.
Happy-path test The everything-went-right case — valid input, expected output. test_signup_success_sets_cookie.
Edge / failure test Deliberately wrong or extreme input — you assert it fails correctly. test_signup_duplicate_email_returns_409, test_signup_long_password_returns_422.

These are not separate categories — a regression test is usually also a unit or integration test. "Regression" just says why it exists.

3. The Test Pyramid

How many of each kind should you write? The classic guideline is the test pyramid: lots of fast unit tests at the base, fewer integration tests in the middle, a handful of slow E2E tests at the top.

Top — E2E (few) Slow, realistic, fragile. A few to confirm the whole system hangs together.
Middle — Integration (some) Endpoints + DB + services. Catches wiring bugs. Moderately fast.
Base — Unit (many) One function each. Milliseconds. This is where most of your tests live.

The reasoning is purely practical:

  • Speed. You run the suite constantly. Unit tests keep it fast enough to run on every save.
  • Precision. When a unit test fails, you know exactly which function broke. When an E2E test fails, the cause could be anywhere.
  • Stability. E2E tests break for boring reasons (a slow network, a renamed button). Too many and people start ignoring red — the worst outcome.
Rule of thumb If a piece of logic can be tested as a unit, test it as a unit. Reach for an integration test only when the thing you care about is the interaction between parts.

4. Choosing the Right Test

A good test suite is not just "more tests". It is the right kind of test for the risk you are trying to reduce. Before writing a test, identify the behaviour, the user or caller who depends on it, and the cheapest layer that can prove it.

If the risk is...Prefer...Because...
A pure function returns the wrong value Unit test Fast, precise, easy to cover many cases.
An API contract breaks Integration test You need the route, validation, service, auth, and database wiring.
A reusable React component behaves wrong Component test You can render just that component and test visible behaviour.
A full workflow breaks across screens E2E test The bug may live in routing, API calls, browser behaviour, or backend state.
A layout or chart changes unexpectedly Visual regression test Normal assertions are weak at catching visual differences.
A fixed bug might come back Regression test The test documents the bug and locks the fix in place.

The cost ladder

Start with the cheapest test that proves the behaviour. Move up the ladder only when a lower-level test cannot see the failure.

  1. Static checks — TypeScript, linting, formatting.
  2. Unit tests — pure logic, helpers, validators.
  3. Component tests — rendered UI behaviour.
  4. Integration tests — API, DB, services, providers.
  5. E2E tests — complete user journeys in a browser.
  6. Manual checks — exploratory testing and UX judgment.
Design pressure If a small behaviour can only be tested with a huge E2E test, the code may be too tightly coupled. Testing pain is often architecture feedback.

A reusable test-case template

When adding a new behaviour to this guide later, write it in this shape. It keeps tests readable and makes future sections easy to extend. 

Behaviour: what must be true from the user's or caller's point of view
Layer: unit | component | integration | E2E | visual
Arrange: the smallest setup needed
Act: the one action being tested
Assert: the observable result
Negative cases: what must fail cleanly
Regression risk: any past bug this protects against

5. A Common Testing Stack

A modern full-stack app often has three active test suites: a backend suite, a frontend component/unit suite, and a Playwright E2E suite. They use different tools, but they follow the same testing discipline: small setup, one clear action, and an observable assertion.

Backend stack

PieceWhat it does
pytest The test runner. It finds every file named test_*.py, runs every function named test_* inside, and reports pass/fail.
pytest-asyncio Lets pytest run async test functions — needed because some backend functions are asynchronous.
assert Plain Python. assert x == y — if the condition is false, the test fails. No special library.
TestClient From FastAPI/Starlette. A fake HTTP client that calls your app in-process — real requests and responses, no real network, no running server.
SQLite test database Tests do not touch the real Postgres database. They use a throwaway SQLite database that is created empty and destroyed after each test.
tests/conftest.py A special pytest file holding shared fixtures (reusable setup — see section 7).

Frontend stack

PieceWhat it does
Vitest Runs JavaScript, TypeScript, and React tests. A common frontend script is npm run test, which runs vitest run.
React Testing Library Renders React components and lets tests query the DOM by roles, labels, text, and other user-facing signals.
user-event Simulates realistic user actions such as typing, clicking, keyboard input, and selection.
jsdom Provides a browser-like DOM inside Node so component tests can run without opening a real browser.
jest-dom Adds readable DOM assertions such as toBeInTheDocument(), toBeVisible(), and toBeDisabled().

E2E stack

PieceWhat it does
Playwright Runs browser tests against the real app. A common root script is npm run test:e2e, which runs playwright test.
playwright.config.js Defines the E2E test directory, browser project, base URL, timeouts, global setup, and the backend/frontend dev servers Playwright starts before tests run.
e2e/fixtures.js Shared E2E setup. Fixtures can create a project, sign in, upload a sample CSV, and pass useful values such as projectId into a test.
data-testid A stable selector used when a control has no good accessible role/name or when visible text is likely to change. Prefer role/label/text first when they are clear.
Postgres The E2E backend talks to a real Postgres database. Local runs need a valid DATABASE_URL, often postgresql://postgres:postgres@localhost:5432/example_app when using a local Docker Compose database.

Where backend tests live

Backend tests usually live under backend/tests/, with one file roughly per area of the app:

  • test_transformations.py — unit tests for the data-transformation logic.
  • test_auth.py — integration tests for sign-up, sign-in, logout, protected routes.
  • test_new_features.py — a mix: unit tests for transforms, integration tests for project endpoints.
  • test_save_revert.py, test_user_logs.py — service- and endpoint-level tests for saving and history.
  • test_save_project.py — a regression test guarding the original file.
  • test_upload.py — unit tests for upload-file validation.
  • conftest.py — shared fixtures, not tests themselves.

Where frontend tests live

Frontend tests live next to the code they protect, usually inside __tests__/ folders under frontend/src/. Examples include:

  • src/Components/common/__tests__/Button.test.jsx — reusable button behaviour.
  • src/Components/common/__tests__/Modal.test.jsx — modal visibility, accessibility, and close interactions.
  • src/api/__tests__/client.test.js — API client behaviour.
  • src/hooks/__tests__/useForm.test.ts — custom hook state and validation behaviour.

Where E2E tests live

E2E tests live in the root e2e/ directory. Each file is a user-facing area or journey:

  • auth.spec.js — sign-in and sign-up screens.
  • project-crud.spec.js — create, open, list, and delete projects.
  • table-operations.spec.js — editing, context menus, rows, and columns.
  • transformations.spec.js — browser-level transform flows.
  • checkpoints.spec.js — save and revert checkpoint journeys.
  • export.spec.js — export modal and downloaded file checks.
  • fixtures/ — sample files used by tests.
Why SQLite for backend tests The real app runs on Postgres. Tests use SQLite because it needs no server, starts instantly, and can be created fresh and thrown away per test — giving every test a clean, isolated database. The trade-off: SQLite and Postgres are not identical, so a few behaviours (some constraints, some SQL features) can differ. For an MVP that trade-off is fine; just be aware of it. Playwright E2E tests are different: they use the real backend and therefore need Postgres.

6. Anatomy of a Test

Almost every good test follows the same three-beat shape, often called Arrange – Act – Assert.

  1. Arrange — set up the inputs and the world the test needs.
  2. Act — run the one thing you are actually testing.
  3. Assert — check the result is what you expected.

A real unit test from test_transformations.py:

def test_cast_to_integer(self):
    # Arrange: build a small input table
    df = pd.DataFrame({"val": ["10", "20", "30"]})

    # Act: run the one function under test
    result = cast_data_type(df, "val", "integer")

    # Assert: state exactly what should be true
    assert result["val"].dtype == "Int64"
    assert result.iloc[0]["val"] == 10

And a real integration test from test_auth.py:

def test_signup_success_sets_cookie(self, anon_client):
    # Act: send a real HTTP request through the app
    resp = anon_client.post(
        "/auth/signup",
        json={"email": "newuser@test.com", "password": "testpassword"},
    )

    # Assert: status code, and the auth cookie came back
    assert resp.status_code == 201
    assert "access_token" in resp.headers.get("set-cookie", "")

The naming convention

Test names are sentences. test_signup_duplicate_email_returns_409 tells you the scenario and the expected result before you read a single line. When that test fails, the name alone often tells you what broke. Long, descriptive names are good here — this is the one place verbosity pays off.

Test classes

You will see tests grouped inside class TestSignup:, class TestCastDataType:, and so on. That is purely organisational — a folder for related tests. pytest does not require classes; some projects use them to keep related cases together.

Checking that something fails correctly

Sometimes the correct behaviour is to raise an error. You test that with pytest.raises — the test passes only if the expected exception is raised: 

def test_non_csv_rejected(self):
    file = MockUploadFile("data.xlsx")
    with pytest.raises(HTTPException, match="not allowed"):
        await validate_upload_file(file)

This asserts: "uploading a non-CSV must raise an HTTPException whose message contains 'not allowed'". If no exception is raised, the test fails — silently accepting a bad file would be a bug.

7. Fixtures

Most tests need some setup before they can run: a database, a logged-in user, a sample file. Copy-pasting that setup into every test would be miserable. A fixture is pytest's answer — a reusable, named piece of setup.

You define a fixture once in conftest.py. Any test that wants it just lists its name as a function argument, and pytest runs the fixture and passes the result in. That argument wiring is the mechanism — no imports, no calls. 

@pytest.fixture
def test_user(db):
    """Create and return a user row for tests that need an owner."""
    user = models.User(email=TEST_USER_EMAIL, password_hash=TEST_USER_PASSWORD_HASH)
    db.add(user)
    db.commit()
    db.refresh(user)
    return user


# A test "asks for" the fixture simply by naming it:
def test_checkpoint_message(self, db, test_user):
    project = create_project(db, name="test", file_path="/tmp/test.csv",
                             description="test", owner_id=test_user.id)
    ...

Example fixtures

FixtureGives the test…
db A database session connected to the throwaway SQLite test database.
test_user A real user row already saved in the database — an owner for projects.
client A TestClient that is already logged in as test_user. Use it for normal endpoint tests — the whole API requires auth.
anon_client A TestClient with no login. Use it to test sign-up/sign-in, and to confirm protected routes reject anonymous callers.
sample_csv A small CSV file on disk, ready to upload.

Two fixtures run automatically

conftest.py has two autouse=True fixtures — they run for every test without being asked for:

  • setup_database — creates all database tables before the test and drops them after. This is what makes every test isolated: each one starts with an empty database and leaves nothing behind.
  • _bypass_app_lifespan — switches off the real app's start-up steps (connecting to Postgres, running migrations) which would otherwise fire when the TestClient starts.
A real test-harness gotcha The client fixture attaches the auth token as a Cookie header rather than through the HTTP cookie jar. The jar mangles cookies for dot-less hostnames (the test host testserver becomes testserver.local) and silently drops them — which made every authenticated test fail with 401. The lesson: when a test fails, suspect the test harness, not just the code under test.

8. Different Things, Different Tests

This is the part you asked about directly: different parts of the code call for different testing methods. You do not test a pure calculation the way you test an HTTP endpoint. Here is how to think about each layer of an application.

Pure logic → unit tests

Functions that take inputs and return outputs with no side effects (no DB, no files, no network). The transformation functions — cast_data_type, rename_column, drop_na.

Method: hand them data, assert on the result. No fixtures, no client. Test many cases — they are cheap. See test_transformations.py.

API endpoints → integration tests

The routes — POST /projects/upload, GET /projects/recent. Here you care about the HTTP contract, not internal details.

Method: use TestClient to send a request; assert on the status code, the response body, and any headers (e.g. the auth cookie).

Database code → integration tests

Functions that read or write rows — create_project, create_checkpoint. A mismatch with the real schema only shows up against a real database.

Method: use the db fixture (throwaway SQLite), call the function, then query the database back and assert the row is as expected.

Auth & ownership → negative tests

Security code is defined by what it refuses. Testing only the success case proves nothing.

Method: test every state — logged in, logged out (401), and "logged in as the wrong user" (404). See TestOwnership in test_auth.py.

API testing in more depth

Since the backend is mostly an API, API tests carry a lot of weight. For any endpoint, walk this checklist:

CheckExample assertion
Correct status code on success assert resp.status_code == 201
Response body has the right shape / values assert resp.json()["email"] == "newuser@test.com"
Secrets are not leaked assert "password_hash" not in resp.json()
Bad input is rejected with the right code assert resp.status_code == 422 (invalid email)
Duplicate / conflict handled assert resp.status_code == 409 (email taken)
Auth is enforced assert resp.status_code == 401 (no cookie)
You cannot reach another user's data assert resp.status_code == 404 (not the owner)

Regression tests → lock in a fix

When a bug is found, the durable fix has two parts: correct the code, and write a test that would have caught it. That test is a regression test. test_save_project.py is a good example — a past concern was that saving a project might overwrite the original uploaded file, so the test uploads, saves repeatedly, and asserts the original file's contents are byte-for-byte unchanged. The bug now cannot return unnoticed.

9. Frontend Component Tests

Frontend tests use the same core idea as backend tests — run code, check the result — but the question changes. Backend tests usually ask: "given this input, did the function, API, or database flow produce the right output?" Frontend component tests ask: "given this screen state, can the user see the right thing, do the right action, and get the right result?"

The mental shift is simple and important: test behaviour, not implementation. Do not test whether a component uses useState, what its internal variable is called, or which helper function it calls. Test the promise the component makes to the user or to its parent.

What a component test checks

QuestionExample
Does it render the right thing? A button with text Save appears as a real button.
Does it respond to user actions? Clicking Save calls the onClick handler.
Does it handle states? Loading, empty, error, disabled, selected, open, closed.
Is it accessible enough to find like a user would? Find buttons by role, inputs by label, dialogs by role.
Does it notify the parent correctly? A form calls onSubmit with the values the user typed.

The frontend testing stack

These frontend examples use Vitest, React Testing Library, @testing-library/user-event, and jsdom. Vitest runs the tests. jsdom gives the test a fake browser-like DOM. React Testing Library renders components into that DOM. user-event simulates realistic clicks, typing, keyboard presses, and other user actions.

The thought process

  1. What is this component responsible for? A button should render, support disabled state, and call onClick. A modal should appear when open, disappear when closed, expose a dialog role, and close through supported interactions.
  2. What can the user observe? Prefer screen.getByRole("dialog") over checking an internal isOpen state value.
  3. What can the user do? Use realistic actions: click the button, type into the input, press Escape.
  4. What outcome matters? Did text appear? Did the button become disabled? Did validation show? Was the callback called with the right value?
Frontend rule If the user could not see it, click it, type into it, focus it, or experience its result, be suspicious of testing it directly.

A realistic component example

frontend/src/Components/common/__tests__/Button.test.jsx tests the reusable Button component. The most important test is not about Tailwind classes; it is about the user's action and the component's contract: 

import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import Button from "../Button";

it("calls onClick handler when clicked", async () => {
  const user = userEvent.setup();
  const handleClick = vi.fn();

  render(<Button onClick={handleClick}>Click</Button>);

  await user.click(screen.getByRole("button"));

  expect(handleClick).toHaveBeenCalledOnce();
});

Read that test as a sentence: render a button, click it like a user, and expect the click handler to run once. It does not care how the component is implemented internally.

Bad vs good assertions

A brittle frontend test reaches into implementation details: 

expect(container.querySelector(".submit-button")).not.toBeNull();

A stronger test asks for the element the way a user or assistive technology would understand it: 

expect(screen.getByRole("button", { name: /submit/i })).toBeEnabled();

Common component checklists

Form

Renders fields, accepts input, shows validation errors, disables invalid submit, and calls onSubmit with the right values.

Modal

Renders only when open, exposes role="dialog", shows the right content, closes from the close button, and closes on Escape if that is part of the contract.

List or table

Shows rows accurately, handles loading, empty, and error states, and supports any visible sorting, filtering, or pagination controls.

Dropdown or menu

Opens on the expected action, shows options, selects the right option, closes afterward, and handles keyboard behaviour when important.

Where E2E and visual tests fit

Component tests are not the whole frontend story. Use end-to-end tests when the important behaviour is a full user journey across routing, frontend, backend, and database — for example upload a CSV, transform it, save it, and reopen it. Use visual regression tests when the main risk is layout or appearance changing unexpectedly, especially for dashboards, tables, charts, and shared design-system components.

10. Industry-Standard Examples

The patterns below are adapted from common tool documentation and production testing practice. Treat them as examples of the shape of a good test, not code you must paste exactly into any one project.

Vitest: small deterministic logic

Vitest's basic model is the same as pytest: define a named test, run code, and assert the result. This style is ideal for helpers, validators, formatters, and transformation logic. 

import { expect, test } from "vitest";

function formatColumnName(value) {
  return value.trim().toLowerCase().replaceAll(" ", "_");
}

test("formats a column name for API payloads", () => {
  expect(formatColumnName(" Total Sales ")).toBe("total_sales");
});

Notice the properties of this test: no DOM, no network, no app provider, no database. If the logic is pure, keep the test pure.

React Testing Library: accessible component behaviour

Testing Library pushes you toward queries that match how users and assistive technology understand the page. For example, getByRole("button", { name: /save/i }) is stronger than searching for a CSS class because it verifies that a real accessible button exists. 

import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";

function RenameColumnForm({ onSubmit }) {
  return (
    <form
      onSubmit={(event) => {
        event.preventDefault();
        const form = new FormData(event.currentTarget);
        onSubmit(form.get("name"));
      }}
    >
      <label>
        New column name
        <input name="name" />
      </label>
      <button type="submit">Save</button>
    </form>
  );
}

test("submits the typed column name", async () => {
  const user = userEvent.setup();
  const onSubmit = vi.fn();

  render(<RenameColumnForm onSubmit={onSubmit} />);

  await user.type(screen.getByLabelText(/new column name/i), "Revenue");
  await user.click(screen.getByRole("button", { name: /save/i }));

  expect(onSubmit).toHaveBeenCalledWith("Revenue");
});

This test has the right shape: render the component, perform real user actions, and assert the observable contract.

Playwright: full browser journey

Playwright tests run in a real browser. They are useful when a bug could involve routing, browser APIs, network calls, storage, authentication, or multiple screens. Playwright's locators also use user-facing queries such as role and accessible name.

The basic syntax is always the same: 

import { test, expect } from "./fixtures.js";

test("plain-English behaviour name", async ({ page, projectId }) => {
  // Arrange: navigate or use a fixture-created project.
  await page.goto(`/projects/${projectId}`);

  // Act: do one user-visible action.
  await page.getByRole("button", { name: /save checkpoint/i }).click();

  // Assert: wait for the user-visible result.
  await expect(page.getByText(/checkpoint saved/i)).toBeVisible();
});

This should not replace smaller tests. It proves that the real workflow hangs together; it does not need to cover every edge case of CSV parsing.

Playwright fixtures: what { page, projectId } means

The argument object passed to a test contains fixtures. page is the browser tab. A custom fixtures.js file can also provide values such as projectId, which means the test starts with an authenticated user and a real sample project already created. Use those fixtures instead of repeating sign-up, upload, and setup steps inside every test. 

test("renames a column", async ({ page, projectId }) => {
  await page.goto(`/projects/${projectId}`);
  // test starts from a real uploaded project, not an empty app
});

Locator syntax: how to find elements

A locator is Playwright's handle for an element. Locators auto-wait before actions and assertions, which is why they are better than manually sleeping. Pick the selector that describes how a user understands the control:

Best when...Use thisExample
The element is a button, link, menu item, heading, table, cell, etc. getByRole page.getByRole("button", { name: /export/i })
An input has a visible label. getByLabel page.getByLabel(/email/i).fill("a@b.com")
You need visible text that is not a button label. getByText page.getByText(/project saved/i)
The element has a placeholder and no better label. getByPlaceholder page.getByPlaceholder(/search/i).fill("age")
The UI text is unstable, duplicated, icon-only, or hidden in a custom control. locator('[data-testid="..."]') page.locator('[data-testid="toolbar-export"]')

Avoid selectors like .blue-button, div:nth-child(3), or long DOM chains. They test the page's implementation instead of the user's behaviour and break during harmless refactors.

Common Playwright actions and assertions

// Navigate
await page.goto("/");
await page.goto(`/projects/${projectId}`);

// Click
await page.getByRole("button", { name: /export/i }).click();

// Type into inputs
await page.getByLabel(/project name/i).fill("Sales cleanup");
await page.getByLabel(/project name/i).press("Enter");

// Upload a file
await page.getByLabel(/csv file/i).setInputFiles("e2e/fixtures/sample.csv");

// Assert visible UI
await expect(page.getByRole("heading", { name: /sales cleanup/i })).toBeVisible();
await expect(page.getByText(/failed to export/i)).not.toBeVisible();

// Assert attributes or state
await expect(page.locator('[data-testid="export-format-csv"]'))
  .toHaveAttribute("aria-pressed", "true");

Waiting: do not sleep

Good E2E tests wait for real signals. Do not use waitForTimeout(1000) unless you are debugging. It makes tests slow and still flaky. Instead, wait for the element, URL, response, or download that proves the app moved forward. 

// Good: waits until the UI is actually ready
await expect(page.getByRole("table")).toBeVisible();
await expect(page).toHaveURL(/\/projects\/[a-f0-9-]+/);

// Good: wait for a specific control before clicking
await page.locator('[data-testid="toolbar-export"]').waitFor({ state: "visible" });
await page.locator('[data-testid="toolbar-export"]').click();

// Bad: guesses that one second is enough
await page.waitForTimeout(1000);

Downloads, dialogs, and other event races

Browser events can happen immediately after a click. Create the promise before the click that triggers the event. In a typical export flow, clicking the toolbar button opens a modal; the download does not happen until the modal's Export button is clicked. 

test("export downloads a valid CSV file", async ({ page, projectId }) => {
  await page.goto(`/projects/${projectId}`);

  await page.locator('[data-testid="toolbar-export"]').click();
  await expect(page.locator('[data-testid="export-filename"]')).toBeVisible();
  await expect(page.locator('[data-testid="export-format-csv"]'))
    .toHaveAttribute("aria-pressed", "true");

  const downloadPromise = page.waitForEvent("download", { timeout: 15_000 });
  await page.locator('[data-testid="export-confirm"]').click();
  const download = await downloadPromise;

  expect(download.suggestedFilename().toLowerCase()).toMatch(/\.csv$/);
});

What makes a good E2E test

  • It covers a real user journey. Good: upload CSV, apply transform, export result. Too small for E2E: pure CSV parsing.
  • It has one main behaviour. A test named "export downloads a valid CSV file" should not also test checkpoint revert and column rename.
  • It uses stable locators. Prefer role, label, text, then data-testid. Avoid CSS classes and DOM position.
  • It waits for outcomes, not time. Use expect(...).toBeVisible(), toHaveURL(), waitForEvent(), or a response wait.
  • It stays independent. Do not rely on another test running first. Use fixtures to create the project or user state you need.
  • It asserts what matters. Check the downloaded CSV header/content, not that a specific internal React state changed.

Cypress: selectors and realistic app testing

Cypress's public guidance emphasizes realistic app testing, independent tests, and stable selectors. For E2E tests, a dedicated data-testid or Testing Library query is usually more stable than selecting by styling classes. 

describe("project list", () => {
  it("opens an existing project", () => {
    cy.visit("/");

    cy.findByRole("link", { name: /recent projects/i }).click();
    cy.findByRole("button", { name: /sales cleanup/i }).click();

    cy.findByRole("heading", { name: /sales cleanup/i }).should("be.visible");
  });
});

Async UI: wait for the user-visible result

Async tests should wait for the thing the user sees, not for an arbitrary timeout. In component tests, that usually means findBy.... In Playwright, it usually means an auto-waiting assertion such as toBeVisible(). 

// Component test
expect(await screen.findByText(/project saved/i)).toBeInTheDocument();

// Browser test
await expect(page.getByText(/project saved/i)).toBeVisible();

11. The Thought Process

Tools aside, the real skill is deciding what to test. When you write or change a function, ask three questions.

1. What is the happy path?

The ordinary, everything-valid case. Sign up with a good email and password → expect 201 and a cookie. This is the first test — if the normal case is broken, nothing else matters.

2. What are the edge cases?

The boundaries and unusual-but-legal inputs. For a password length rule of 8–72 bytes: test 7 (too short), test 73 (too long), and ideally the exact boundaries. Bugs cluster at boundaries — that is where < versus <= mistakes hide.

3. What are the failure cases?

The things that should go wrong, and must fail cleanly. Duplicate email → 409. No auth cookie → 401. Wrong owner → 404. A function that only works when given perfect input is not finished, and the failure tests are what prove the other half.

One behaviour per test Each test should check a single behaviour and its name should say which. When it fails, the name alone points at the problem. A test that asserts ten unrelated things just tells you "something, somewhere".

Test behaviour, not implementation

Assert on what a function produces, not how. A test tied to internal details breaks every time you refactor — even when behaviour is unchanged — which trains people to ignore failures. Check the observable result.

Write the failing test first

When fixing a bug, it is worth writing the test before the fix. Watch it fail — that proves the test actually exercises the bug. Then fix the code and watch it pass. If you write the test after, you never confirmed it could catch anything. (Doing this for all code, not just bugfixes, is called test-driven development — optional, but the "see it fail first" habit is gold.)

What not to test

  • Other people's code. You do not test that FastAPI routes requests or that pandas adds numbers — that is their job. Test your logic.
  • Trivial code. A one-line getter with no logic has nothing to verify.
  • Everything at once. 100% coverage is rarely worth the effort. Aim tests at logic that is complex, important, or easy to get wrong — auth, transformations, money-like invariants.

12. Running the Tests

Backend commands often run from the backend/ directory. Frontend commands run from the frontend/ directory. E2E commands run from the repository root because playwright.config.js is there.

Run the backend suite

uv run pytest

uv run executes the command inside the project's virtual environment, so the right Python and packages are used.

Run the frontend suite

npm run test

This runs vitest run, which executes the React and JavaScript tests once and exits. The frontend test setup lives in frontend/src/test/setup.js, where @testing-library/jest-dom adds useful assertions like toBeInTheDocument() and toBeDisabled().

Run the E2E suite

npm run test:e2e

This runs Playwright from the repo root. Playwright starts the backend and frontend dev servers from playwright.config.js. The backend needs Postgres. If your local database is the Docker Compose one, run: 

DATABASE_URL=postgresql://postgres:postgres@localhost:5432/example_app npm run test:e2e

Useful backend variations

# Quiet output (one character per test)
uv run pytest -q

# Run a single file
uv run pytest tests/test_auth.py

# Run a single test class or function
uv run pytest tests/test_auth.py::TestSignup
uv run pytest tests/test_auth.py::TestSignup::test_signup_success_sets_cookie

# Stop at the first failure
uv run pytest -x

# Show print() output and full detail
uv run pytest -s -v

Useful frontend variations

# Run one test file
npx vitest run src/Components/common/__tests__/Button.test.jsx

# Run tests whose name matches a pattern
npx vitest run -t "calls onClick"

# Run in watch mode while developing
npx vitest

Useful E2E variations

# Run one E2E file
npm run test:e2e -- e2e/export.spec.js

# Run tests whose name matches a pattern
npx playwright test -g "export downloads"

# Run headed so you can see the browser
npm run test:e2e:headed

# Debug one file interactively
npm run test:e2e:debug -- e2e/export.spec.js

Reading the result

The last line is usually the summary. Green is the goal:

177 passed, 2 failed, 128 warnings in 5.79s
  • passed — these behaviours are confirmed working.
  • failed — pytest prints, for each, the file, line, and the exact assert that did not hold. Start there.
  • warnings — not failures; usually deprecations worth cleaning up eventually.

A failure block shows the comparison directly — e.g. assert 401 == 404 means the code returned 401 where the test expected 404. That one line is usually enough to start debugging. Vitest failures look similar: they name the failing file and test, then show the expected value versus the received value.

Tests are not the only check Alongside tests, the repo uses linters and formatters: ruff for Python, and ESLint / Prettier for the frontend. Tests check that the code is correct; linting and formatting check that it is clean and consistent. Both should be green before code is considered done.
Green is not "bug-free" A passing suite means "every behaviour I thought to check still works". It cannot vouch for cases nobody wrote a test for. Tests raise your confidence; they never make it absolute.

When you add a feature, write tests for…

  • the happy path — valid input, expected output;
  • each edge case — boundaries, empty values, unusual-but-legal input;
  • each failure case — bad input, missing auth, wrong owner;
  • and, if you fixed a bug, one regression test that fails without your fix.
